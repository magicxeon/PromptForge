# Phase 2-10 Durable Jobs and Batch Orchestration

**Status:** Proposed - Awaiting Review

## 1. Business Requirement

Fashion packages may contain multiple paid operations and image jobs. Work must survive server restart, support pause/resume/retry and settle credits correctly per operation.

## 2. Domain Model

```text
GenerationPlan
  -> Batch
      -> Billable Operations
          -> Jobs
              -> Attempts
              -> Results/Events
```

Batch states:

```text
draft -> quoted -> confirmed -> queued -> processing
processing -> paused | partially_completed | completed | failed | cancelled
```

Job states:

```text
pending -> claimed -> running -> succeeded
                          -> retry_wait -> pending
                          -> failed | cancelled
```

## 3. Reliability

- Persist jobs before acknowledging submission.
- Use leases/heartbeats so abandoned running jobs can recover.
- Claim jobs atomically.
- Define maximum attempts and exponential backoff with jitter.
- Cancellation is cooperative and records whether provider cost already occurred.
- Successful jobs are never regenerated by resume.
- Provider request IDs and attempt metadata are retained for support.

## 4. Billing Integration

- Confirmed Batch references reservation and immutable quote.
- Each operation settles or releases according to outcome policy.
- Infrastructure/provider failure is distinguished from user-requested regeneration.
- Safety rejection uses explicit refund/retry policy.
- Job worker never changes balance without LedgerService.

## 5. Events and UX

- Persist authoritative events; SSE is a delivery mechanism, not the source of truth.
- Reconnecting clients fetch current Batch snapshot then resume events.
- Show overall and per-product progress.
- Pause stops new claims but does not corrupt running attempts.
- Retry failed only creates attempts for eligible failed operations.

## 6. Acceptance Criteria

- Restart during processing does not lose or duplicate accepted jobs.
- Pause/resume and retry failed only behave deterministically.
- Duplicate submission with same idempotency key returns the same Batch.
- Per-job credit settlement reconciles with Batch summary.
- Existing queue behavior migrates without losing supported OpenAI/Gemini streaming.
- Concurrency, lease recovery and cancellation are covered by tests.
