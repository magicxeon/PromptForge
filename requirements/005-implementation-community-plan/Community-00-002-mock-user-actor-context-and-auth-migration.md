# Community-00-002 Mock User, Actor Context and Auth Migration

**Status:** Proposed - Awaiting Review  
**Feature type:** Identity foundation and development user switcher  
**Depends on:** Application shell, server middleware, local JSON storage  
**Created:** 2026-07-19

## 1. Business Requirement

Community needs cross-user behavior before real authentication exists. We must be able to test:

- Alice owns a generated result.
- Alice shares a template.
- Bob remixes it.
- Bob cannot reuse Alice private face reference.
- Admin hides or removes a public post.

This requires a mock user system that acts like production auth from the domain-service point of view.

## 2. System Design

### 2.0 Source Of Truth And Data Origin

This requirement is the source of truth for **development identity** until production authentication is implemented.

Use these sources:

```text
User identity source:
  server/identity/mockUsers.json

Current active user source in browser:
  localStorage key: mpf_active_mock_user_id

Request actor source:
  HTTP header: x-mpf-user-id

Server actor source:
  actorContextMiddleware resolves x-mpf-user-id through MockUserRepository

Future production source:
  Supabase Auth session -> ActorContext
```

Do not derive ownership from:

```text
client username text field
X-User-Role
image metadata username
history item username only
Community post display name
```

Those values may exist for backward compatibility, but they are not authorization boundaries.

### 2.1 Actor Context Schema

```text
ActorContext
- userId: string
- username: string
- displayName: string
- role: user | creator | admin | support
- activeCreatorProfileId: string | null
- isMockActor: boolean
- authProvider: mock | supabase | system
- requestId: string
- originalRequesterUserId: string | null
```

Rules:

- `userId` is the durable identity key.
- `username` is compatibility/display only.
- Server creates `ActorContext`; client never directly grants permissions.
- Services receive `ActorContext`, not raw headers.
- Background jobs use `system` actor plus original requester id.
- `requestId` is generated by middleware when missing.
- `originalRequesterUserId` is null for normal web requests and set for background/system operations.

### 2.2 Mock User JSON Schema

```text
MockUser
- schemaVersion: number
- id: string
- username: string
- displayName: string
- role: user | creator | admin | support
- activeCreatorProfileId: string | null
- creditBalanceSeed: number
- status: active | disabled
- featureFlags: string[]
- createdAt: ISO string
- updatedAt: ISO string
```

Seed users:

```text
usr_demo   -> username: user_demo, role: user
usr_alice  -> username: user_alice, role: creator
usr_bob    -> username: user_bob, role: user
usr_admin  -> username: admin_demo, role: admin
```

### 2.3 Auth Migration Rule

When Supabase Auth is added:

```text
Supabase session -> ActorContext -> same domain services
```

Do not rewrite Community services to know Supabase directly.

### 2.4 Compatibility With Current App State

The current application already has legacy user concepts:

```text
state.username
state.userRole
/api/credits?user=...
server/database.json mock users/credits
history records with username
```

Migration compatibility rule:

- Keep legacy fields working during this phase.
- Introduce `ActorContext` as the new service boundary.
- When both `userId` and `username` are present, `userId` wins.
- When only `username` exists, resolve to a mock user if possible.
- Do not change unrelated Scene Builder naming or handoff behavior as part of this requirement.

## 3. Software Development Specification

### Client Files

```text
client/core/actorContext.js
client/core/apiClient.js
client/community/communityMockUserSwitcher.js
client/shell/userMenu.js
client/index.html
client/app.js
```

### Client File Responsibilities

#### `client/core/actorContext.js`

Owns browser-side active mock user state.

Required API:

```text
getActiveMockUserId() -> string
setActiveMockUserId(userId: string) -> void
getActorHeader() -> { "x-mpf-user-id": string }
subscribeActorChange(listener) -> unsubscribe
```

Behavior:

- Reads/writes localStorage key `mpf_active_mock_user_id`.
- Defaults to `usr_demo`.
- Emits an event when active user changes.
- Does not store permissions or decide ownership.

#### `client/core/apiClient.js`

Central wrapper for API calls that need actor headers.

Required API:

```text
apiFetch(url, options) -> Promise<Response>
apiJson(url, options) -> Promise<any>
```

Behavior:

- Adds `x-mpf-user-id` from `actorContext`.
- Adds JSON headers when body is a plain object.
- Keeps existing direct `fetch` calls working during migration, but new Community code should use `apiFetch/apiJson`.

#### `client/community/communityMockUserSwitcher.js`

Development-only UI control.

Behavior:

- Loads available mock users from `GET /api/mock-users`.
- Shows active mock user display name.
- Lets developer switch between demo/alice/bob/admin.
- Calls `setActiveMockUserId`.
- Triggers refresh hooks for Community/History/Shared Templates panels.
- Is hidden when `window.MPF_ENABLE_MOCK_USERS === false` or server says mock users are disabled.

#### `client/shell/userMenu.js`

Optional integration point for showing active actor in the shell.

If the shell does not have a user menu yet, the switcher may render in a small development-only container in `index.html`.

### Server Files

```text
server/identity/mockUsers.json
server/identity/MockUserRepository.js
server/identity/mockActorContext.js
server/middleware/actorContextMiddleware.js
server/server.js
test/mockActorContext.test.js
```

### Server File Responsibilities

#### `server/identity/mockUsers.json`

Stores canonical mock users using the schema in section 2.2.

Rules:

- Keep small and deterministic.
- Do not store secrets.
- Do not store real user email/password.

#### `server/identity/MockUserRepository.js`

Repository API:

```text
listActiveUsers() -> Promise<MockUser[]>
findById(userId) -> Promise<MockUser | null>
findByUsername(username) -> Promise<MockUser | null>
toActorContext(mockUser, requestId) -> ActorContext
```

#### `server/identity/mockActorContext.js`

Pure functions:

```text
createSystemActorContext(originalRequesterUserId, requestId) -> ActorContext
createAnonymousDeniedContext(requestId) -> null
normalizeActorContext(input) -> ActorContext
```

Use this for tests and background jobs.

#### `server/middleware/actorContextMiddleware.js`

Middleware behavior:

```text
read x-mpf-user-id
if missing in development -> usr_demo
if missing in production -> unauthenticated / reject mutating request
resolve MockUser
if user disabled -> reject mutating request
attach req.actorContext
attach req.requestId
```

The middleware must not trust `x-user-role` or client-supplied role headers.

#### `server/server.js`

Required route wiring:

```text
app.use(actorContextMiddleware)
GET /api/mock-users
GET /api/me
```

`GET /api/mock-users` returns sanitized user list:

```text
id
username
displayName
role
activeCreatorProfileId
featureFlags
```

It must not return internal credit seeds as authoritative balances once credit ledger exists.

### API Headers During Development

```text
x-mpf-user-id: usr_alice
x-mpf-request-id: optional-debug-request-id
```

### API Contracts

#### `GET /api/me`

Returns current actor:

```json
{
  "userId": "usr_alice",
  "username": "user_alice",
  "displayName": "Alice Creator",
  "role": "creator",
  "activeCreatorProfileId": "creator_alice",
  "isMockActor": true,
  "authProvider": "mock"
}
```

#### `GET /api/mock-users`

Returns mock switcher options:

```json
{
  "enabled": true,
  "users": [
    {
      "id": "usr_alice",
      "username": "user_alice",
      "displayName": "Alice Creator",
      "role": "creator"
    }
  ]
}
```

### Existing Endpoint Migration Requirements

Any endpoint touched during this requirement should begin accepting `req.actorContext`.

Minimum endpoints to inspect:

```text
/api/generate
/api/history
/api/credits
/api/scene-templates/share-drafts
/api/scene-templates/publish
/api/scene-templates/shared
/api/scene-templates/remix-events
```

For first pass:

- Do not rewrite all endpoints fully.
- Add actor context without breaking existing `username` payloads.
- New Community code must use actor context.
- Existing legacy calls can continue until phased migration.

## 4. Implementation Plan

### 4.1 In Scope

This requirement implements the identity foundation only:

```text
mock user data
actor context middleware
client active-user helper
API fetch wrapper
development user switcher
basic /api/me and /api/mock-users
tests for actor resolution
```

### 4.2 Out Of Scope

Do not implement these here:

```text
renaming Story Mode to Scene Builder
rewriting cross-mode handoff
credit ledger deduction
community feed UI
creator profile UI
payment
Supabase Auth
production session cookies
```

If those files must be touched only to pass actor headers, keep the change minimal and document why.

### 4.3 Step-by-Step Work

1. Create `server/identity/mockUsers.json` using the canonical schema.
2. Create `MockUserRepository`.
3. Create `mockActorContext` helper functions.
4. Create `actorContextMiddleware`.
5. Register middleware and routes in `server/server.js`.
6. Create `client/core/actorContext.js`.
7. Create `client/core/apiClient.js`.
8. Create `client/community/communityMockUserSwitcher.js`.
9. Add switcher script and container to `client/index.html`.
10. Add switcher initialization in `client/app.js`.
11. Update touched Community/Scene Template API calls to use `apiFetch/apiJson`.
12. Add tests.

### 4.4 Implementation Acceptance Checklist

Before marking this requirement complete, verify:

```text
GET /api/me returns usr_demo by default in development.
GET /api/me returns usr_alice when x-mpf-user-id=usr_alice.
Unknown user id is rejected for mutating requests.
Disabled mock user is rejected for mutating requests.
Client switcher persists selected user in localStorage.
Client API wrapper sends x-mpf-user-id.
Server domain services can read req.actorContext.
No authorization rule trusts username or X-User-Role alone.
Existing generation/history flow still works with usr_demo.
```
